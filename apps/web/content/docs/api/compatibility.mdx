title: Compatibility
description: Which methods and options each adapter supports, where behaviour differs, and where a feature falls back or is unavailable.
---

Every adapter implements the same unified surface, but providers differ in what they support natively. This matrix is the canonical per-adapter reference for where each method works directly, where it works with a caveat, and where it has no equivalent at all.

Each cell is one of three states — hover any **caveat** or **throws** cell for the specifics:

- **Supported** — the method maps to a native provider primitive.
- **Supported with caveat** — it works, but with a behavioural difference worth knowing: a client-side fallback (`copy` as read-then-write, say), a buffered stream, an ignored option, or a tighter limit.
- **Throws** — the provider has no primitive for it, so the method fails loudly rather than silently misbehaving. Reach for the [escape hatch](/features/escape-hatch) when you need the feature anyway.
